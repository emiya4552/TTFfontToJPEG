# TTF 项目开发规范

本文件适用于 `TTF/main` 目录及其所有子目录。

## 头文件与源文件

- 头文件与 `.c` 文件必须分开。
- `.h` 文件只放公共数据类型、公共宏和函数声明。
- `.c` 文件放具体实现、内部函数、内部常量，以及仅实现所需的头文件。
- 仅在当前 `.c` 文件中使用的函数应声明为 `static`，不要暴露不必要的接口。

## 代码风格

- 保持原项目已有的 C 代码风格。
- 命名、缩进、括号、变量声明和对齐方式尽量与相邻代码一致。
- 不以现代化或格式统一为理由大规模格式化旧代码。
- 所有代码注释必须使用中文。
- 每个函数前必须使用统一的 `// 功能：...` 注释说明函数用途。
- 较长的条件判断、循环和连续逻辑块前必须添加简短中文注释。
- 注释用于解释代码目的，不逐行重复代码本身。

## 修改原则

- 非必要情况下，不修改或重写已有代码。
- 增加功能时优先创建新函数，并通过新函数调用、组合和复用旧函数。
- 只有旧接口无法支持新功能时，才进行最小范围的接口调整。
- 不顺带修复与当前任务无关的细枝末节问题。

## 库与功能实现

- 优先使用 C 语言标准库，例如 `stdio.h`、`stdlib.h`、`string.h` 和 `stdint.h`。
- 不随意引入第三方库。
- TTF 表解析、字符串拆分、排版、数组管理等数据结构和核心功能应自行实现。
- 不使用 FreeType、HarfBuzz、stb_truetype 等字体库代替项目的核心实现。
- 必须使用平台能力时，只在必要范围内使用，例如 Windows 编码转换和 Linux `iconv`。

## 修改范围与测试

- 当前再开发只修改 `TTF/main`，不要修改旧项目目录。
- 不修改与当前任务无关的 JPEG 代码。
- 不覆盖或删除已有字体、BMP 和可执行文件。
- 编译、测试和临时输出应放在隔离的临时目录中，并在验证完成后清理。
